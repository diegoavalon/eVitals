# Implementation Handoff: eHealth Core Web Vitals Dashboard

## Goal

Build a greenfield static reporting dashboard for daily Lighthouse results across the pages configured in `urls.config.json`.

The dashboard should help engineering/performance owners quickly identify failing pages, group issues by page metadata, inspect trends, and open the full Lighthouse HTML report from a row-level side drawer.

## Final architecture

Use a static GitHub Pages architecture:

- Source code, config, tests, and runner scripts live on `main`.
- GitHub Actions runs Lighthouse daily and on manual dispatch.
- Generated dashboard data, Lighthouse JSON, Lighthouse HTML, and static site files persist on the `gh-pages` branch.
- GitHub Pages serves from `gh-pages`.
- Public GitHub Pages publication is acceptable.
- No always-on Express/Node server is needed.

Use:

- npm
- Vite
- React
- TypeScript
- Tailwind CSS only where compatible with the provided design system
- CSS variables for light/dark theme tokens, aligned to the provided design system
- The provided root `DESIGN.md` as the binding UI/design specification
- The provided `app/components/ui` branded component library, which wraps `@base/ui`, as the required component source
- Vitest for core logic tests
- Lightweight inline SVG for sparklines and summary donut charts
- Pinned local Lighthouse CLI dependency

Recommended React tech stack:

| Area                 | Recommendation                                                                                  |
| -------------------- | ----------------------------------------------------------------------------------------------- |
| App framework        | **Vite + React + TypeScript**                                                                   |
| Hosting model        | **Static GitHub Pages**, generated by GitHub Actions                                            |
| Routing              | **React Router with HashRouter** for GitHub Pages-safe navigation                               |
| UI system            | **`DESIGN.md` + `/ui` branded components first**, direct `@base/ui` only when no wrapper exists |
| Styling              | **Tailwind CSS + design-system CSS variables**                                                  |
| State                | Local React state + URL params; avoid global state unless needed                                |
| Data loading         | Static JSON fetched from `data/dashboard.json` and run manifests                                |
| Validation           | **Zod** for `urls.json`, `dashboard.config.json`, and generated manifest schemas                |
| Tests                | **Vitest + React Testing Library**                                                              |
| Runner scripts       | Node + TypeScript via **tsx**                                                                   |
| Lighthouse execution | Pinned local `lighthouse` npm dependency                                                        |
| Process helpers      | `execa` for CLI execution, `p-limit` for concurrency                                            |
| Charts               | Inline SVG React components; avoid charting library for v1                                      |
| CI/CD                | GitHub Actions scheduled + `workflow_dispatch`, publish to `gh-pages`                           |

## Page configuration

`urls.config.json` remains the page registry for v1.

Required fields:

```json
{
  "id": "homepage",
  "label": "Homepage",
  "url": "https://www.ehealthinsurance.com/",
  "group": "core"
}
```

Validation requirements:

- `id` is required and unique.
- `label` is required.
- `url` is required and must be a valid URL.
- `group` is required.
- Do not require new fields in v1.

## Dashboard configuration

Add `dashboard.config.json` on `main`.

Recommended shape:

```json
{
  "defaultCategory": "performance",
  "enabledCategories": [
    "performance",
    "accessibility",
    "best-practices",
    "seo"
  ],
  "devices": ["mobile", "desktop"],
  "historyLimit": 30,
  "basePath": "/ehi-perf-dashboard-v2/"
}
```

The workflow/build scripts should read this config and expose relevant values through generated static JSON for the client.

The Vite base path must be configurable with `VITE_BASE_PATH`, defaulting to `/ehi-perf-dashboard-v2/`.

## GitHub Actions workflow

Create a Lighthouse dashboard workflow with:

- `schedule`: `0 14 * * *`
- `workflow_dispatch`
- source checkout from `main`
- generated output checkout/restore from `gh-pages`
- npm install/build/test steps
- Chrome/Chromium verification
- Lighthouse run step
- data generation step
- retention pruning
- publish/push to `gh-pages`

Timezone note:

- GitHub Actions cron is UTC-based.
- `0 14 * * *` matches 9:00 AM America/Chicago during daylight time.
- Do not implement DST-aware scheduling in v1.

Chrome policy:

- Use `ubuntu-latest`.
- Verify the preinstalled Chrome/Chromium path.
- Set `CHROME_PATH`.
- Fail clearly for infrastructure/config errors if Chrome is unavailable.
- Do not bundle Chromium via Playwright/Puppeteer in v1.

Workflow failure policy:

- Page/device Lighthouse failures after retry should not fail the workflow.
- Publish partial run data with failed entries visible in the dashboard.
- Fail the workflow only for infrastructure/config errors such as invalid config, missing Chrome, broken build, or inability to publish.

## Lighthouse runner

Implement a Node runner script invoked by GitHub Actions.

Behavior:

- Read `urls.json`.
- Read `dashboard.config.json`.
- Generate tasks for every `{page, device}` pair.
- Run all Lighthouse categories for both mobile and desktop.
- Invoke the pinned local Lighthouse CLI binary, not a global install.
- Use a single workflow job with concurrency limit `2`.
- Timeout each page/device attempt after `5 minutes`.
- Retry each failed page/device task once.
- Preserve both JSON and HTML output for every successful page/device task.
- Record failed page/device tasks in the run manifest.

Device behavior:

- Mobile and desktop both ship in v1.
- Mobile should use Lighthouse mobile behavior.
- Desktop should use Lighthouse desktop behavior, e.g. the CLI desktop preset.

## Generated artifact layout

Persist generated content on `gh-pages`.

Use this layout:

```text
/
  index.html
  assets/
  data/
    dashboard.json
    runs/
      2026-06-02T14-00-00Z/
        manifest.json
  reports/
    runs/
      2026-06-02T14-00-00Z/
        homepage.mobile.report.json
        homepage.mobile.report.html
        homepage.desktop.report.json
        homepage.desktop.report.html
```

Retention:

- Keep the latest 30 run folders by default.
- Make the limit configurable.
- Prune both `data/runs/*` and `reports/runs/*` consistently.

## Static data loading

The React app should fetch static JSON files from GitHub Pages at runtime.

Minimum generated files:

- `data/dashboard.json`: current dashboard model, run index, enabled categories, enabled devices, latest selected defaults, and enough history for sparklines.
- `data/runs/{runId}/manifest.json`: complete run details for a run.

Do not parse full Lighthouse JSON files in the browser for the main dashboard. Full JSON remains published for debugging/future detail views.

## Manifest/result model

Each page/device result should include:

- `runId`
- `fetchTime`
- `pageId`
- `label`
- `url`
- `group`
- `device`
- `status`: `good`, `needs-improvement`, `failing`, or `run-failed`
- `reportJsonPath`
- `reportHtmlPath`
- Lighthouse category scores for all enabled categories
- Performance metrics:
  - LCP
  - CLS
  - TBT
  - performance score
- failure details when Lighthouse fails

## Status rules

Performance focused view:

- LCP good: `<= 2.5s`
- LCP needs improvement: `> 2.5s` and `<= 4.0s`
- LCP failing: `> 4.0s`
- CLS good: `<= 0.1`
- CLS needs improvement: `> 0.1` and `<= 0.25`
- CLS failing: `> 0.25`
- TBT good: `<= 200ms`
- TBT needs improvement: `> 200ms` and `<= 600ms`
- TBT failing: `> 600ms`
- A page passes Performance only if LCP, CLS, and TBT are all good.
- Page Performance status is the worst metric status.
- Lighthouse performance score is displayed with standard Lighthouse score color bands separately.

Non-Performance focused views:

- Use selected Lighthouse category score bands:
  - good: `>= 90`
  - needs improvement: `50-89`
  - failing: `< 50`
- This applies to Accessibility, Best Practices, and SEO.

Partial run behavior:

- A failed page/device Lighthouse task counts in the denominator as unavailable/failing.
- Median score calculations use successful reports only.
- Failed rows should be visible and clearly marked.

## Dashboard UI behavior

Match the provided `index-mockup.jpg` direction while adapting for static hosting.

Strict UI implementation contract:

- A `DESIGN.md` file will be provided in the project root and must be strictly followed.
- A branded component library will be provided in `/ui`; it wraps `@base/ui` and should be the first-choice component source.
- Use `/ui` branded components whenever a suitable branded component exists.
- If a needed component is not yet available in `/ui`, import directly from `@base/ui` rather than hand-rolling an equivalent component.
- Do not import from `@base/ui` directly when a suitable branded `/ui` wrapper exists.
- Do not hand-roll components that are available in `/ui`.
- Custom UI should be limited to thin composition of `/ui` components, design tokens, and layout needed for dashboard-specific behavior.
- If `index-mockup.jpg` and `DESIGN.md` conflict, `DESIGN.md` takes precedence.

Global controls:

- Device toggle: Mobile/Desktop.
- Category focus control near the device toggle.
- Default category comes from `dashboard.config.json`, initially Performance.
- Theme toggle: functional light/dark mode with persistence, e.g. localStorage.
- Run check button: link to the GitHub Actions workflow dispatch page. It must not attempt to trigger Actions directly from the static site with an embedded token.

Device toggle:

- Filters the entire dashboard to one selected device at a time.
- Summary counts, priority card, rows, sparklines, and drawer report all reflect only the selected device.

Category focus:

- Shows one focused Lighthouse category at a time to avoid clutter.
- Performance view shows Core Web Vitals-centric metrics and pass logic.
- Accessibility, Best Practices, and SEO views use category score status and should not clutter rows with every category at once.

Home view:

- Summary headline:
  - For Performance: count pages where LCP/CLS/TBT are all good out of configured pages for the selected device/latest run.
  - For other categories: count pages whose selected category score is good out of configured pages for the selected device/latest run.
- Score card:
  - For Performance: median Lighthouse performance score across successful page reports.
  - For other categories: median selected category score across successful page reports.
- Include pass/needs-improvement/failing counts.
- Include a "Needs attention first" card.
- Include a "Most Recent" report list resembling the mockup.

Priority card:

- For Performance: choose the latest selected-device page with the worst status, then sort by number of failing metrics, severity over thresholds, and lower Lighthouse performance score.
- For non-Performance categories: choose the latest selected-device page with the worst selected-category status, then lower selected-category score as tie-breaker.

Rows:

- Show page label, group, selected device, focused category score/status, and relevant metrics.
- In Performance view, show LCP, CLS, and TBT values/bars.
- In non-Performance views, show selected category score and status without adding all category columns.
- Row sparkline shows selected device LCP history over retained runs.
- Row delta compares latest LCP to the previous run for the same page/device.

Side drawer:

- Opens when a report row is clicked.
- Renders the saved Lighthouse HTML report in an iframe.
- iframe `src` should be the relative static path from the selected row, e.g. `reports/runs/{runId}/{pageId}.{device}.report.html`.
- Do not inject raw Lighthouse HTML into the React DOM.

All Pages view:

- Functional in v1.
- Provide grouped/filterable table view by page group, device, category focus, and status.

Navigation:

- Home: functional.
- All Pages: functional.
- Trends: disabled/coming soon.
- Settings: disabled/coming soon.

## Definition of done

The handoff is complete when a coding agent can:

- Initialize the Vite/React/TypeScript app.
- Follow root `DESIGN.md` and implement UI using the provided `/ui` branded component library.
- Add functional dark mode using the provided design system tokens/components.
- Validate `urls.json` and `dashboard.config.json`.
- Run Lighthouse for all configured pages across mobile and desktop.
- Generate JSON and HTML artifacts with stable page/device paths.
- Build `data/dashboard.json` and per-run manifests.
- Render the Home dashboard from static JSON.
- Render the All Pages view.
- Open saved Lighthouse HTML in a side drawer iframe.
- Publish the generated static site/reports to `gh-pages`.
- Run on GitHub Actions via schedule and manual dispatch.
- Keep only the latest 30 runs by default.
- Pass tests and production build.

## Remaining implementation assumptions

- The repository will be created/configured with `main` and `gh-pages` branches.
- GitHub Pages project URL will use `/ehi-perf-dashboard-v2/` unless `VITE_BASE_PATH` overrides it.
- The GitHub Actions workflow dispatch URL can be configured into generated dashboard data or derived from repository metadata.
- Public Lighthouse artifacts are acceptable because the user confirmed public GitHub Pages is acceptable.
